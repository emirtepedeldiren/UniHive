Can someone explain the time complexity of QuickSort in the worst-case scenario?

I understand that the average case is O(n log n), but I'm getting confused about why it degrades to O(n^2) when the array is already sorted. Isn't the pivot choice supposed to prevent this? If we always pick the last element as the pivot in a sorted array, doesn't it just partition it into an empty array and an array of size n-1? Any visual explanations would be super helpful. Trying to wrap my head around this before the midterm!

You hit the nail on the head with the partitioning part!

When the array is already sorted, and you consistently pick the last element as the pivot, you are creating the most unbalanced partitions possible.

Array: [1, 2, 3, 4, 5]
Pivot: 5
Partition 1: [1, 2, 3, 4] (size n-1)
Partition 2: [] (size 0)

This means instead of dividing the problem in half (which gives you the log n depth), you are only reducing the problem size by 1 at each recursive step. So you make n recursive calls, and each call does O(n) work to partition. Summing that up: n + (n-1) + (n-2) ... = O(n^2).

Just to add to what David said, this is exactly why modern implementations of QuickSort rarely use the naive "pick the last element" strategy.

Usually, people use something like "Median of Three" (picking the median of the first, middle, and last elements) or randomized pivot selection to probabilistically guarantee that you don't hit that O(n^2) worst case on already sorted or reverse-sorted data.
